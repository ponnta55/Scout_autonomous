## 3.1 何を解いたか

> **LiDAR 点群だけで、リアルタイムに「自分が今どこにいるか」を推定する** こと。
> ホイールエンコーダや IMU には頼らない (頼れない) 構成を目指した。

---

## 3.2 採用: KISS-ICP

### 3.2.1 概要

**KISS-ICP** (Keep It Small and Simple ICP) — 連続する LiDAR スキャンを ICP マッチングし、相対変位を累積して自己位置を推定するアルゴリズム。

```
1 フレームの処理:

  入力: 生 LiDAR 点群
       ↓
  ① Deskew         (スキャン中のロボット動きを補償)
       ↓
  ② Voxelize       (点群を間引いて計算量を削減)
       ↓
  ③ ICP            (前フレームの局所マップに位置合わせ)
       ↓
  ④ マップ更新      (今のフレームを局所マップに追加)
       ↓
  出力: pose (x, y, z, roll, pitch, yaw)
```

### 3.2.2 採用理由

| 比較対象 | KISS-ICP の優位 |
|---------|----------------|
| LOAM / LIO-SAM | IMU 不要、依存少 |
| Cartographer | 軽量、チューニングパラメータが少ない |
| AMCL + 既知地図 | 事前マップが要らない、屋外でも動く |

**「最小の道具立てで動かす」** という方針に最も合致したため採用。

### 3.2.3 採用上のトレードオフ (隠さず書く)

| 弱点 | 影響 | 緩和策 |
|------|------|--------|
| ループクロージャなし | 長距離走行で drift が累積 | 屋内短距離運用では実用上 OK。将来 GNSS 統合 |
| 純回転時に死ぬことがある | KISS が publish 停止する瞬間がある | 上位で `/odometry/filtered` の停止を検知 → 安全停止 |
| 静的環境を仮定 | 動的物体 (歩行者) が多いと ICP が引っ張られる | 大半が静的なら影響軽微。voxel 平均化で滑らかに |

## 3.3 EKF 構成の進化 (Before / After)

このプロジェクトの **「データ駆動で設計を変えた」** 部分です。

### 3.3.1 Before — wheel + KISS の EKF 融合

```
[wheel] /odom ──┐
                ├─→ EKF ─→ /odometry/filtered → 下流ノード
[KISS-ICP] /kiss/odometry ─┘
```

**設計意図**:
- 短期: wheel が高精度に角速度を提供
- 長期: KISS-ICP が絶対 yaw を補正
- 両者を融合して drift 抑制

### 3.3.2 評価実験 (2026-05-21)

純回転試験 (720° × 2 = 1440° の回転) を 5 ラン実施し、3 つの odometry をすべて CSV に記録:

- `/odom` (wheel)
- `/kiss/odometry` (KISS)
- `/odometry/filtered` (EKF 融合)

#### 観察 1: **EKF が KISS にほぼ完全追従**

| ラン | KISS yaw drift | EKF yaw drift | 差 |
|------|----------------|---------------|------|
| 1812 | -3.49° | -3.62° | 0.13° |
| 1825 | +1.24° | +1.25° | 0.01° |
| 1827 | +0.78° | +0.73° | 0.05° |

EKF は実質 KISS のスムージング層になっていた。

#### 観察 2: **「wheel が小さい drift」の罠**

wheel が報告する yaw drift は ±0.4° と非常に小さく、一見すると「wheel の方が KISS より高精度」のように見えた。
しかしこれは校正バイアス由来の **見かけの drift** であって、絶対精度ではない (詳細 §3.4)。

#### 観察 3: **KISS-ICP の死亡再現性**

ラン 1802 で KISS が回転開始 5 秒で publish 停止 → 純回転耐性が確率的。

![3 ソース yaw 比較 (校正前)](../assets/images/slam_three_sources_yaw.png)

![3 ソース XY 軌跡 (校正前)](../assets/images/slam_three_sources_xy.png)

> グラフ: wheel (青)・KISS (緑)・EKF (赤) の比較。EKF が KISS にほぼ重なっているのが視覚的にも明らか。

### 3.3.3 After — KISS 単独構成

```
/kiss/odometry ─→ EKF ─→ /odometry/filtered ─→ 下流ノード
                  ↑
              (将来) GNSS / IMU
```

#### 切替の判断

| 案 | 採否 | 理由 |
|----|------|------|
| EKF を完全削除 | ✗ | 将来 GNSS/IMU 統合時に再構築コスト |
| scout_base の校正値を直接修正 | ✗ | SDK が隠蔽していて変更困難 |
| wheel\_corrector を維持して EKF に流す | ✗ | EKF が活かしきれない (§3.4) |
| **wheel 入力を切り、KISS 単独に** | **✓** | シンプル・データに基づく |

#### 切替の利点と欠点

| 項目 | 旧 (融合) | 新 (KISS 単独) |
|------|----------|---------------|
| 純回転時の挙動 | KISS drift をそのまま | 同左 |
| KISS 死亡時 | wheel で dead-reckoning (バイアス入り) | EKF も止まる (**明確な失敗信号**) |
| wheel バイアスの影響 | あり | **なし** |
| デバッグ容易性 | 中 | **高** |

**「複雑な融合 < シンプルな単一ソース」** — これは多くの SLAM プロジェクトでよく見る "EKF wisdom" の逆ですが、データが示した結論に従いました。

## 3.4 Wheel Odometry のキャリブレーションバイアス

> 詳細は別ドキュメント (社内) にも残してあります。ここでは要点のみ。

### 3.4.1 発見

校正なし (wheel\_yaw\_scale = 1.0) で純回転テストを実施:

| 観察対象 | 値 |
|---------|-----|
| 物理回転 (目視) | 約 810° |
| wheel 報告 | 720° |
| **wheel / 物理** | **0.889** |

→ **wheel が回転量を約 11% 過小報告** している。

### 3.4.2 原因究明

| 容疑 | 判定 | 理由 |
|------|------|------|
| EKF | 無罪 | wheel が返した値を信じるだけ |
| `/cmd_vel` 指令経路 | 無罪 | 標準的な publish 経路 |
| エンコーダ物理 | 無罪 | 直線走行は正常 |
| **scout\_base のキネマティクス** | **有罪** | track\_width パラメータが実物より大きい |
| **SCOUT MINI firmware** | **共犯** | 同じ track\_width を使用 |

#### 因果関係

```
[ユーザー指令] /cmd_vel.angular.z (yaw rate)
       ↓
[scout_base] track_width を使って wheel 角速度に変換
       ↓  ← track_width が実物より約 12% 大きい
[firmware] その wheel 角速度通りにモーター制御
       ↓
[物理] 想定より速く回る (= 報告より多く回っている)
       ↓
[エンコーダ] wheel 回転を正しく計測
       ↓
[scout_base] 同じ track_width で yaw を逆算
       ↓
[/odom] コマンド値とほぼ同じ値を報告 (= 実回転と乖離)
```

### 3.4.3 暫定対処

`wheel_odom_corrector` ノードで `/odom` をリスケーリングし `/odom_corrected` として再 publish 可能。
ただし最終的には **wheel を EKF から外す** ことで根本回避。

## 3.5 TF の統一

### 3.5.1 問題

KISS-ICP の出力 pose は **LiDAR フレーム** で表現される。一方、planner や controller は **base\_link** を基準に動く。この不整合があると:

- 「ロボットの前方 X m」の解釈がコンポーネント間でずれる
- 安全半径 (ロボット形状) の中心がどこか曖昧
- ROI の方向と LiDAR の方向が食い違う

### 3.5.2 解決方針

| 設計判断 | 内容 |
|---------|------|
| **基準フレーム** | すべて `base_link` (ロボットの幾何中心) |
| **LiDAR フレーム** | `base_link` からの `static TF` で位置決め |
| **KISS-ICP 出力の扱い** | LiDAR フレームでの delta を base\_link に変換して累積 |
| **planner / controller** | すべて base\_link 座標で動作 |

これにより:

- LiDAR の物理取付位置を変えても、static TF を更新するだけで他に波及しない
- 計画経路と障害物検出が同じ座標系で扱える
- デバッグ時に「どのフレームの値か」で迷わない

## 3.6 評価結果のサマリー (詳細は §06)

純回転試験 3 ラン平均:

| 指標 | 値 |
|------|-----|
| KISS yaw drift (720° × 2 = 1440° 回転後) | < 3.5° |
| KISS drift rate | 0.05 – 0.24% |
| XY drift final | 30 – 55 mm |
| trial-to-trial 標準偏差 (yaw) | ~2.6° |
