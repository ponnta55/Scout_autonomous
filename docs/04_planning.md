# 04. 時空間経路計画

[← 03. SLAM](03_slam.md) | [次: 05. 歩行者検出 →](05_pedestrian.md)

---

## 4.1 何を解いたか

> **動的環境 (歩行者がいる) で、「待機 vs 迂回」を自動で選び、衝突せずに目的地に到達する経路を計画する。**

---

## 4.2 静的 2D 計画の限界

従来の 2D グリッド A\* / Nav2 の計画は:

```
時刻 = 「今」だけを見る
  ↓
歩行者は「現在位置」に block
  ↓
動いている歩行者の進行方向は無視される
```

これでは:

- 歩行者がロボットの目の前を **通り過ぎる** 場合 → 過剰迂回
- 歩行者の **進行方向** と交差する場合 → 衝突 or 凍結
- **「数秒待てば通れる」** という意思決定ができない

実環境ではこの動作で日常的に問題が起きます。

## 4.3 STP4 — `(x, y, t)` 探索

### 4.3.1 中心アイデア

**時間軸を計画空間に追加する。**

```
2D 計画: グリッドセル (x, y) を「占有 / 自由」で分類
3D 計画: グリッドセル (x, y, t) を「占有 / 自由」で分類
                                 ↑
                             時間軸を追加
```

### 4.3.2 コストマップの構築

歩行者の予測軌跡を時間軸方向にスタック:

```
時刻 t = 0:  歩行者は位置 P_0 → P_0 周辺を inflate して block
時刻 t = 1:  歩行者は P_1 にいる予測 (= 0.1 秒先) → P_1 を block
時刻 t = 2:  P_2 を block
...
時刻 t = k:  P_k を block (k = horizon)
```

→ **時間軸 t を持つ 3D グリッド** の (x, y, t) セルが「占有 / 自由」

### 4.3.3 探索

```
start = (x_0, y_0, t = 0)
goal  = (x_g, y_g, t = 任意)

neighbor として展開:
  ・ (x±1, y±1, t+1)  ← 移動 (ロボットの速度上限以内)
  ・ (x,   y,   t+1)  ← wait (その場で待つ)

最小コスト経路を A* 風探索で見つける
```

`wait` アクションを許可することで **「待機」が選択肢として浮上** します。これが時空間計画の核心。

### 4.3.4 出力

```
[(x_0, y_0, t_0),
 (x_1, y_1, t_1),
 ...
 (x_g, y_g, t_g)]
```

各セルが時刻情報を持つので、ロボットはどの時刻にどこにいるべきかが明示的に分かります。

---

## 4.4 利点

| 項目 | 効果 |
|------|------|
| **歩行者を未来の障害物として表現** | 衝突予測が可能 |
| **wait アクション** | 待機戦略が自動で選ばれる |
| **時間軸での評価** | 動的環境で stable |
| **既存のグラフ探索** | A\* の枠組みでそのまま使える |

---

## 4.5 計算コスト

| 項目 | 値 (典型) |
|------|----------|
| グリッドサイズ | W × H × T (W, H = 数十、T = 数十) |
| 計画レイテンシ | **5 – 50 ms** |
| 計画頻度 | 5 Hz (200 ms 周期) |

5Hz 周期内で完結。**GPU 不要**。

---

## 4.6 実機展開で見えた問題と対処

ここからは「実機に乗せたら出てきた泥臭い問題」の話。

### 4.6.1 計画失敗時の暴走

**現象**: 計画が失敗 (障害物に囲まれた / コストマップエラー) すると、ロボットは停止すべきところで **前回の target\_heading に向かって回転継続** していた。

**原因**:

```cpp
if (plan_ok) {
  target_heading_ = ...  // 更新
  ...
} else {
  robot_.hold_position();  // 停止指令
  // ← target_heading_ は前回値のまま!
}

publish_cmd_vel();  // 古い heading に向かって回り続ける
```

**対処**: 計画失敗時に `target_heading_ = slam_heading_` (現在向きを維持) にリセット。

### 4.6.2 計画タイムアウト時の永久停止

**現象**: WA\* がタイムアウトすると `allow_partial = true` の動作で「現在位置だけのパス」が返り、ロボットが進まなくなる。

**原因**: パスが `[(start, t=0), (start, t=1)]` だとセグメント長 = 0 で前進距離が 0。

**対処**: タイムアウト時はパスを「無効」扱いし、前回有効なパスがあれば再利用、無ければ停止。

### 4.6.3 経路計画の不連続性

**現象**: 5Hz の計画ごとに微妙に異なる経路が出ることがあり、ロボットの挙動がカクつく。

**対処**: **パスキャッシュ + 妥当性検証** を導入。

```
新しい計画が出るたびに:
  1. キャッシュされた前回パスの妥当性を検証
     (a) 経過時間が短い (age < threshold)
     (b) 現在位置がパスから大きく逸脱していない
     (c) パスが新規障害物で塞がれていない
  2. 妥当ならキャッシュを継続使用、無効なら新パスに切替
```

これで「ガタつき」が大幅に減り、走行が滑らかになりました。

### 4.6.4 ウェイポイントが後方にある場合

**現象**: SLAM のズレでロボットがウェイポイントを通り過ぎたとき、サブゴールが後方に設定され、後ろ向きに経路を計画しようとする。

**対処**: ウェイポイントの方向検証 (ロボットの向きと反対方向のものはスキップ判定)。

### 4.6.5 SLAM 未初期化時の暴走

**現象**: SLAM が起動する前に `publish_cmd_vel` が呼ばれると、`slam_heading_ = 0.0` (初期値) と `target_heading_ = ウェイポイント方向` が乖離し、ロボットが回転を始める。

**対処**: `slam_available_` フラグで gating。SLAM が valid pose を返す前は cmd\_vel を抑止。

## 4.7 タイミング設計

```
時間 (ms)    0       100     200     300     400
             |-------|-------|-------|-------|
パイプライン  計画           計画           計画    (5Hz)
LiDAR        ●   ●   ●   ●   ●   ●   ●   ●       (10Hz)
ホイールodom  ●●●●●●●●●●●●●●●●●●●●●●●●●●●●●●●●●  (高頻度)
cmd_vel      ↑               ↑               ↑
```

- 計画周期 200 ms の間、cmd\_vel は前回の経路を追従
- LiDAR フレームは 2 個ごとに 1 回計画される

### 200 ms ギャップの議論

「100 ms 後に新しい障害物が出たらどうなる?」 → 検知は次の計画 (100 ms 後) まで遅れる。
しかし:

- ロボット速度 0.3 – 0.5 m/s × 0.1 s = 3 – 5 cm の進み
- 安全マージン (0.9 m) と比べれば十分小さい
- 高速運用 (> 1 m/s) のときに再検討が必要

## 4.8 座標系まわりの教訓

### ROI がロボット向きを考慮しない問題

コストマップは odom frame の x-y 軸に沿った矩形で構築していた。これだと:

- ロボットが 90° 回転していると「前方 10m」のつもりが「右方 10m」になる
- ROI の縁で障害物が **見えたり消えたり** する

→ ロボット向きを考慮した ROI に修正する余地あり (今後の課題)。

## 4.9 まとめ

| 観点 | 結論 |
|------|------|
| アルゴリズム | `(x, y, t)` 3D 探索 + wait アクションで動的環境対応 |
| 性能 | 5 – 50 ms / 計画、CPU 動作 |
| 実機特有の問題 | 計画継続性・失敗時挙動・座標系統一 |
| 解決 | パスキャッシュ、heading リセット、TF 統一 |

時空間アルゴリズム自体は研究室の先行成果ですが、**「これが実機で安定して動くようにするにはどうするか」** という運用周りの作り込みが本プロジェクトの主な貢献です。
