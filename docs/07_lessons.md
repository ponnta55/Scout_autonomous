## 7.1 自己点群問題 — 「ロボットが自分を障害物と認識する」

### 7.1.1 現象

シミュレーションで完璧に動いていた構成を実機に持ち込んだ瞬間、ロボットは:

- **走り出さない**
- ときどき **暴走** (自分の周りが「障害物だらけ」と認識し、無理な経路を取る)
- プランナが計算する経路が常にゼロ近傍 (= 待機しか出ない)

「sim で動いていたのに、実機の初日に動かない」**最初の壁**。

### 7.1.2 原因

**実機 LiDAR (Hesai QT128) はロボット自身の構造物を点群として観測する**:

- 自分のシャシ縁
- LiDAR マウントの支柱
- アンテナ・配線
- センサーハウジング

これらが「自分の半径 0.5m 以内に大量の点群」として現れ、障害物検出 → 「自分が完全に塞がれている」と判断 → 待機 or 強引な小経路で暴走。

### 7.1.3 なぜシミュには無かったか

| 環境 | LiDAR の挙動 |
|------|-------------|
| Gazebo `gpu_lidar` | URDF で指定した sensor 位置から仮想ビームを撃つ。ロボット mesh をレイトレースで **通過する設定** (default) |
| 実機 | 当然、自分の体は見える |

→ シミュではロボット自身が「見えない」(壁を貫通する透明体扱い)。
これは Gazebo の仕様で、悪い設定ではないのですが、**「sim で OK = real で OK」を保証しない最大要因** のひとつ。

### 7.1.4 解決: 3D ボックス自己フィルタ

ロボット筐体を LiDAR 中心の 3D ボックスでモデル化し、その中の点を削除。**ただし高さ上限を設ける**:

```
点 (x, y, z) を削除する条件 (AND):

  X (前後): 一定範囲内
  Y (左右): 一定範囲内
  Z (上下):           z ≤ LiDAR + 15cm のみ削除

  → 全条件成立 → 自己点として削除
```

#### 高さ上限が重要

```
ロボット脇に歩行者が立っている状況:

  [上] ──────────────────  ← 歩行者の頭・胴
                                  ↑ ボックス外 → 残る
  [LiDAR + 15cm] ──────────  ← この線が分かれ目
                                  ↓ ボックス内 → 削除
  [LiDAR レベル] ────────
                                  ロボット筐体
  [床] ──────────────────
```

→ ロボットの脚 (床近く) は削除、歩行者の上半身は残る → **隣接歩行者を見落とさない**。

### 7.1.5 検証

| 状態 | 点群数 |
|------|--------|
| フィルタなし | 約 1700 / frame |
| フィルタあり | 600 – 800 / frame |
| 削除率 | ~50% |

歩行者検出への影響なし。

### 7.1.6 一言

> **「sim で動いた」は実機の保証にはならない。最低限、センサーが見るものに対する物理モデルを疑え。**

## 7.2 Wheel Odometry の校正バイアス

### 7.2.1 発見の経緯

純回転試験で wheel と KISS の値を並べて見ていて、**「wheel と KISS で値が違う」** ことに気付いた。
直感的には wheel の方が正確だと思っていたが、目視で物理回転と突き合わせると:

| 観察 | 値 |
|------|-----|
| 物理回転 (目視) | ~810° |
| wheel 報告 | 720° |
| KISS 報告 | 810° |

→ **wheel の方が間違っていた**。約 11% 過小報告。

### 7.2.2 原因究明の流れ

「コマンドと wheel と物理が三角関係でズレている」状態。原因を一つずつ容疑から外していった:

| 容疑 | 結果 | 検証方法 |
|------|------|----------|
| EKF | 無罪 | EKF を bypass しても /odom が同じ値 |
| 指令側 | 無罪 | 標準的な /cmd_vel publish |
| CAN 通信 | 無罪 | 通信レート・ロス共に正常 |
| エンコーダ | 無罪 | 直線走行は正確 (タイヤ径は正しい) |
| **scout_base / firmware の track_width** | **有罪** | 残った |

### 7.2.3 因果関係

```
[指令] /cmd_vel = 0.524 rad/s (左右輪の差から決定する yaw rate)
       ↓
[scout_base] 内部キネマティクスで wheel 速度に変換
       ↓  ← ここで使う track_width が実物より約 12% 大きい
[firmware] その wheel 速度通りに駆動 → 物理は速く回る (~0.589 rad/s)
       ↓
[エンコーダ] wheel 回転を正しく計測
       ↓
[scout_base] 同じ track_width で yaw を逆算 → コマンド値と一致 (0.524)
       ↓
[/odom] 0.524 rad/s として publish (実回転 0.589 と乖離)
```

両端で同じ間違った係数を使っているため、wheel 報告は **「コマンドに忠実」だが「物理と乖離」** という、データだけ見ても気付きにくい状態。

### 7.2.4 解決

**短期**: wheel\_odom\_corrector ノードでスケーリング (`/odom_corrected` を作る)
**根本**: SDK 内の track\_width を実測値に修正 (SDK が隠蔽していて変更困難)
**最終判断**: **EKF から wheel 入力を切る** ([03. SLAM](03_slam.md) §3.3)

### 7.2.5 一言

> **データだけ見て「ホイールは精度が高い」と思い込まないこと。物理測定とセンサー報告の突き合わせは、定期的にやる価値がある。**

## 7.3 計画継続性 — 「ロボットがガタつく」

### 7.3.1 現象

走行中、5Hz で計画される経路が微妙に異なり、ロボットが小刻みに方向を変えてカクつく。

### 7.3.2 原因

```
計画サイクル N:   経路 = [P_0, P_1, P_2, ...]
計画サイクル N+1: 経路 = [P_0', P_1', P_2', ...]   (微妙に違う)
```

毎回の計画でわずかに異なる経路が出るため、target heading が振動。

### 7.3.3 解決

**パスキャッシュ + 妥当性検証**:

```
新しい計画が出るたびに:
  1. キャッシュされた前回パスの妥当性を検証
     - 経過時間が短い (age < 閾値)
     - 現在位置がパスから大きく逸脱していない
     - パスが新規障害物で塞がれていない
  2. 妥当ならキャッシュを継続使用、無効なら新パスに切替
```

これで滑らかさが大幅改善。

### 7.3.4 一言

> **「計算結果が毎回違う」のはアルゴリズムが悪いのではなく、上位ロジックで安定化させるべき問題。**

## 7.4 計画失敗時の挙動

### 7.4.1 現象

障害物に囲まれた瞬間、ロボットが前回の target heading に向かって回転を続ける。

### 7.4.2 原因

計画失敗 → `hold_position()` で停止指令を出すが、**target\_heading\_ 変数が前回値のまま** だった。
`publish_cmd_vel()` は target との差から角速度を計算するので、古い heading に向かって回転継続。

### 7.4.3 解決

計画失敗時に `target_heading_ = current_heading_` (現在向きを維持) にリセット。

### 7.4.4 一言

> **エラー時の「何もしない」は、「前回の状態を維持する」とは違う。明示的に安全状態に遷移させる必要がある。**

## 7.5 LiDAR タイムスタンプ問題

### 7.5.1 現象

Hesai ドライバが publish するタイムスタンプが想定と異なる挙動を取り、TF lookup や ROS bag 録画で時刻不整合が起きた。

### 7.5.2 解決

ドライバの `use_timestamp_type` を変更し、ホスト PC 時刻を基準にする。
これで ROS の他コンポーネントと時刻整合が取れた。

### 7.5.3 一言

> **時刻まわりは、最初に整合性を確認しておくと後の苦労が減る。**

## 7.6 IP アドレスが消える問題

### 7.6.1 現象

PC を再起動すると、LiDAR との通信に使う IP アドレスが消える。
NetworkManager が DHCP で上書きしてしまう。

### 7.6.2 解決

毎回手動で `ip addr add` するか、nmcli で固定 IP プロファイルを作る。
**起動手順書を作っておくと毎日の負担が減る** という地味な教訓。

## 7.7 サブモジュール初期化忘れ

`git clone` だけしてビルドして「動かない…」と数時間溶かす。
サブモジュールがある場合は `git submodule update --init --recursive` を忘れずに。

これは ROS 2 ドライバ系の落とし穴で、HesaiLidar_ROS_2.0 が典型例。

## 7.8 全体としての教訓

### 「Sim → Real ギャップは予想以上に泥臭い」

| 想定 | 実際 |
|------|------|
| アルゴリズムが正しければ動く | 動かない |
| センサーは仕様通りの値を返す | バイアスがある |
| ロボットは想定通りに動く | コマンドと挙動がズレる |
| 設定ファイルが正しければ起動する | 環境変数・ネットワーク・ファームウェアまで揃って初めて起動 |

### 「データ駆動で判断する」

このプロジェクトでの一番大きな設計判断は **「EKF の wheel 入力を切る」** ですが、これは「直感」ではなく **計測実験のデータに基づいて** 決めました。

> **複雑な融合 < シンプルな単一ソース** — データが示すならそれに従う。

### 「観測 → 仮説 → 検証」のサイクル

すべての問題は、

1. **観測** (現象を見る)
2. **仮説** (容疑を絞る)
3. **検証** (実験する)
4. **判断** (採用 or 棄却)

の繰り返しで解いていました。**研究もエンジニアリングも同じ**、というのが個人的な実感です。
