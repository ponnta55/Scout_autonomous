## 2.2 ハードウェア

### 2.2.1 移動プラットフォーム

| 項目 | 値 |
|------|------|
| 機種 | AgileX SCOUT MINI |
| 駆動方式 | 4輪 skid-steer |
| サイズ (LxWxH) | 約 612 × 580 × 245 mm |
| 重量 | 約 23 kg |
| 最大速度 (公称) | 3.0 m/s |
| 実運用速度 | ~0.5 m/s (安全のため) |
| 通信 | CAN (500 kbps) |
| 制御インターフェース | `/cmd_vel` (Twist) |

### 2.2.2 センサー

| 項目 | 値 |
|------|------|
| LiDAR | Hesai QT128 |
| チャンネル数 | 128 |
| 最大測距 | ~20 m (運用) |
| スキャン頻度 | 10 Hz |
| 1 フレーム点群サイズ | 約 230,400 点 |
| 通信 | Ethernet (UDP) |

### 2.2.3 計算機

- ノート PC (CPU のみ、GPU 不要)
- OS: Ubuntu (ROS 2 Jazzy)

> **GPU を使わない設計が重要な制約条件**でした。KISS-ICP・WA\* プランナ・歩行者検出器いずれも CPU で動くよう構築しています。

## 2.4 TF ツリー

```
   odom
    │
    └─→ base_link               ← 全推定値の基準フレーム
         │
         ├─→ hesai_lidar         ← LiDAR (static TF)
         │
         └─→ (各種センサー)
```

### 設計判断: **全フレームを base_link に統一**

SLAM (KISS-ICP) は LiDAR フレームでの相対変位を返しますが、これをそのまま使うと **planner と SLAM の座標系がずれる** という問題があります。本プロジェクトでは:

1. KISS-ICP の出力 (LiDAR frame) を **TF で base_link に変換**
2. Planner / controller は **すべて base_link を基準** に動作
3. ロボット形状 (自己フィルタ、安全半径) も **base_link 中心** で定義

これにより、LiDAR を物理的に動かしても (例: 取付位置を変えても) `static TF` を更新するだけで他コンポーネントに波及しない設計になっています。

> 詳細は [03. SLAM](03_slam.md) の「TF 統一」セクション参照。

## 2.5 データフローと周期

| データ | 発行元 | 周期 | 用途 |
|--------|--------|------|------|
| 生 LiDAR 点群 | Hesai ドライバ | 10 Hz | SLAM 入力、検出入力 |
| `/kiss/odometry` | KISS-ICP | 10 Hz | 自己位置 |
| `/cmd_vel` | pegasus_node | 5 Hz | ロボット制御 |
| 計画経路 | STP4 | 5 Hz | 可視化 + 制御 |
| 歩行者マーカー | 検出パイプライン | 5 Hz | 可視化 + STP4 入力 |

### なぜ 5Hz?

- LiDAR 10Hz の半分 → 計算予算が 2 フレーム使える
- STP4 計画のレイテンシ (典型 5-50 ms) が 200 ms 周期に収まる
- ロボットの最大角速度 (実運用 ~0.55 rad/s) に対し追従可能
- 制御ループのジッタを抑えやすい (低周波の方が安定)

## 2.6 シミュレーション環境 (検証用)

実機展開の前段として、Gazebo Harmonic で同じノード構成を動かしています。

```
┌─────────────────────────┐
│ Gazebo Harmonic          │
│  ・SCOUT MINI URDF       │
│  ・gpu_lidar センサ       │
│  ・pedestrians_moving    │
│    world (動的歩行者)     │
└──────────┬──────────────┘
           │ ros_gz_bridge
           ↓
┌─────────────────────────┐
│ 同じ pegasus_node が     │
│ 動く (実機と差し替えなし) │
└─────────────────────────┘
```

**設計方針**: 同じノードが sim でも実機でも動くようにし、差分を最小化。これにより「sim で OK → 実機で動かない」場合の原因が **環境差** に絞り込める設計になっています。

> Sim と Real のギャップで実際に問題になった項目は [07. Sim → Real で学んだこと](07_lessons.md) を参照。
