# 05. 歩行者検出・追跡・予測

[← 04. 経路計画](04_planning.md) | [次: 06. 評価・結果 →](06_results.md)

---

## 5.1 何を解いたか

> **LiDAR 点群から歩行者を検出し、フレーム間で追跡 (ID 付与) し、近未来 (~1.2 秒) の軌跡を予測する。**
> 出力は時空間プランナ (STP4) の入力になる。

---

## 5.2 パイプライン全体

```
  LiDAR 点群 (10 Hz)
       │
       ▼
  ① 前処理 (地面除去 + 自己フィルタ)
       │
       ▼
  ② DBSCAN クラスタリング
       │
       ▼
  ③ 特徴量抽出 (各クラスタ → 32次元)
       │
       ▼
  ④ Random Forest 分類 (歩行者 vs その他, ONNX)
       │
       ▼
  ⑤ ByteTrack + Kalman フィルタ で追跡 (ID 維持)
       │
       ▼
  ⑥ Trajectron 系 ONNX で軌道予測 (obs_len=8, pred_len=12)
       │
       ▼
  歩行者 (x, y, t) 予測トラジェクトリ
       │
       ▼
  STP4 プランナへ
```

---

## 5.3 各ステージの詳細

### 5.3.1 前処理

**地面除去**: 床面の点群を高さフィルタで切る。簡易だが効果的。

**自己フィルタ** (詳細は [07. Sim → Real で学んだこと](07_lessons.md)):

ロボット自身の点群を 3D ボックスで削除。Z 上限を設けることで、隣接する歩行者の **胴体・頭部は残す**。

```
削除条件 (LiDAR ローカル frame):
  X 方向: 一定範囲内
  Y 方向: 一定範囲内
  Z 上限: LiDAR + 15 cm 以下のみ削除
            ↑ これがあるおかげで、ロボット脇の歩行者は残る
```

### 5.3.2 DBSCAN クラスタリング

点群を密度ベースで塊にまとめる。各塊が「物体候補」になる。

- 屋外の壁や柱も検出されるが、それは正しい (障害物として STP4 に渡る)
- 歩行者かどうかは次の分類ステージで判定

### 5.3.3 特徴量抽出

各クラスタから 32 次元の特徴量を抽出:

- 形状: 高さ・幅・奥行き・縦横比
- 密度: 単位体積あたりの点数
- 強度分布: LiDAR 反射強度の統計量
- 形状の対称性

これらは「人間の体は他の障害物と統計的に違う」という仮定を活かす。

### 5.3.4 Random Forest 分類

ONNX で量子化された Random Forest が CPU で動く。各クラスタを「歩行者 / その他」に二値分類。

| 項目 | 値 |
|------|-----|
| モデル | Hesai QT128 データで再学習した RF |
| 推論コスト | 1 クラスタあたり < 1 ms |
| 検出スコア閾値 | 高め (誤検出を許さない設定) |

> Random Forest の選定理由: 軽量・解釈可能・特徴量設計と相性が良い。Deep Learning (PointPillars 系) も検討対象だが、Hesai 用ラベル付きデータの収集が必要なため、今後の課題。

### 5.3.5 追跡: ByteTrack + Kalman

フレーム間で同じ人物に同じ ID を割り当てる。

- **Kalman Filter** が各歩行者の位置・速度を線形モデルで保持
- **ByteTrack** が「信頼度の高い検出」と「低い検出」を 2 段マッチングし、オクルージョンに強い

ID が安定することで、軌道予測器に渡せる「観測履歴」が作れる。

### 5.3.6 軌道予測: Trajectron 系 ONNX

過去 `obs_len = 8` フレーム (~0.8 秒) の歩行者位置を入力 → 未来 `pred_len = 12` フレーム (~1.2 秒) の位置を予測。

```
入力: 過去 8 ステップの (x, y) 列
       ↓
[ONNX 推論]
       ↓
出力: 未来 12 ステップの (x, y) 列
```

この予測軌跡が STP4 のコストマップを **時間軸方向** に block する。

---

## 5.4 STP4 への接続

```
歩行者 i の予測軌跡: [(x_0, y_0), (x_1, y_1), ..., (x_11, y_11)]
                       ↓
時空間コストマップに反映:
                       ↓
  t=0:  (x_0, y_0) 周辺を block
  t=1:  (x_1, y_1) 周辺を block
  ...
  t=11: (x_11, y_11) 周辺を block
                       ↓
                   STP4 探索
```

**重要**: 各歩行者の **不確実性** (Kalman の共分散) も block 半径に反映可能な設計。

## 5.5 性能とトレードオフ

| 項目 | 値 |
|------|-----|
| 全パイプライン (検出 + 追跡 + 予測) | 10 Hz でリアルタイム |
| 1 フレームあたり計算時間 (典型) | 10 – 30 ms (CPU only) |
| 同時追跡可能歩行者数 | 数十人 (実用上問題ない範囲) |

### 既知の限界

| 項目 | 内容 |
|------|------|
| 静止歩行者 | 動かない人を「障害物」と区別するのが難しい (どちらも STP4 に渡せば OK) |
| 物陰からの出現 | LiDAR が見えていないと検出不可 (物理限界) |
| 走る歩行者 | 軌道予測の線形仮定が崩れる場合あり |

---

## 5.6 設計判断の振り返り

| 判断 | 採用 | 理由 |
|------|------|------|
| 検出に Deep Learning を使うか | 使わなかった (RF) | データ量制約 + CPU 動作の要件 |
| 追跡に ByteTrack を採用 | ✓ | 2 段マッチングで安定 |
| 軌道予測に ONNX | ✓ | フレームワーク非依存・推論速度安定 |
| 検出器を Hesai 専用に再学習 | ✓ | センサー固有の点群密度に最適化 |
