# 05. 歩行者検出・追跡・予測

[← 04. 経路計画](04_planning.md) | [次: 06. 評価・結果 →](06_results.md)

> **実証範囲**: シミュレーション (Gazebo Harmonic) では検出 → 追跡 → 予測 → STP4 への接続まで一通り動作確認済み。**実機での歩行者環境検証は今後の課題**。

---

## 5.1 何を解いたか

> **LiDAR 点群から歩行者を検出し、フレーム間で追跡 (ID 付与) し、近未来 (~4.8 秒) の軌跡を予測する。**
> 出力は時空間プランナ (STP4) の入力になる。

---

## 5.2 パイプライン全体

```
  LiDAR 点群 (10 Hz)
       │
       ▼
  ① 前処理 (地面除去 + 自己フィルタ)
       │
       ▼
  ② DBSCAN クラスタリング
       │
       ▼
  ③ 特徴量抽出 (各クラスタ → 32次元)
       │
       ▼
  ④ Random Forest 分類 (歩行者 vs その他, ONNX)
       │
       ▼
  ⑤ ByteTrack + Kalman フィルタ で追跡 (ID 維持)
       │
       ▼
  ⑥ Trajectron 系 ONNX で軌道予測 (obs_len=8 frames=3.2s, pred_len=12 frames=4.8s)
       │
       ▼
  歩行者 (x, y, t) 予測トラジェクトリ
       │
       ▼
  STP4 プランナへ
```

---

## 5.3 各ステージの詳細

### 5.3.1 前処理

**地面除去**: 床面の点群を高さフィルタで切る。簡易だが効果的。

**自己フィルタ** (詳細は [07. Sim → Real で学んだこと](07_lessons.md)):

ロボット自身の点群を 3D ボックスで削除。Z 上限を設けることで、隣接する歩行者の **胴体・頭部は残す**。

```
削除条件 (LiDAR ローカル frame):
  X 方向: 一定範囲内
  Y 方向: 一定範囲内
  Z 上限: LiDAR + 15 cm 以下のみ削除
            ↑ これがあるおかげで、ロボット脇の歩行者は残る
```

### 5.3.2 DBSCAN クラスタリング

点群を密度ベースで塊にまとめる。各塊が「物体候補」になる。

- 屋外の壁や柱も検出されるが、それは正しい (障害物として STP4 に渡る)
- 歩行者かどうかは次の分類ステージで判定

### 5.3.3 特徴量抽出

各クラスタから 32 次元の特徴量を抽出:

- 形状: 高さ・幅・奥行き・縦横比
- 密度: 単位体積あたりの点数
- 強度分布: LiDAR 反射強度の統計量
- 形状の対称性

これらは「人間の体は他の障害物と統計的に違う」という仮定を活かす。

### 5.3.4 Random Forest 分類

ONNX で量子化された Random Forest が CPU で動く。各クラスタを「歩行者 / その他」に二値分類。

| 項目 | 値 |
|------|-----|
| モデル | Hesai QT128 データで再学習した RF |
| 推論コスト | 1 クラスタあたり < 1 ms |
| 検出スコア閾値 | 高め (誤検出を許さない設定) |

> Random Forest の選定理由: 軽量・解釈可能・特徴量設計と相性が良い。Deep Learning (PointPillars 系) も検討対象だが、Hesai 用ラベル付きデータの収集が必要なため、今後の課題。

### 5.3.5 追跡: ByteTrack + Kalman

フレーム間で同じ人物に同じ ID を割り当てる。

- **Kalman Filter** が各歩行者の位置・速度を線形モデルで保持
- **ByteTrack** が「信頼度の高い検出」と「低い検出」を 2 段マッチングし、オクルージョンに強い

ID が安定することで、軌道予測器に渡せる「観測履歴」が作れる。

### 5.3.6 軌道予測: Trajectron 系 ONNX

過去 `obs_len = 8` フレーム (**3.2 秒**) の歩行者位置を入力 → 未来 `pred_len = 12` フレーム (**4.8 秒**) の位置を予測。
フレームレート 2.5 Hz (0.4 秒/フレーム)。

```
入力: 過去 8 ステップ (3.2 秒) の (x, y) 列
       ↓
[ONNX 推論]
       ↓
出力: 未来 12 ステップ (4.8 秒) の (x, y) 列
```

この予測軌跡が STP4 のコストマップを **時間軸方向** に block する。STP4 の計画 horizon (4.8 秒) と軌道予測 horizon が一致するよう設計。

---

## 5.4 STP4 への接続

```
歩行者 i の予測軌跡: [(x_0, y_0), (x_1, y_1), ..., (x_11, y_11)]
                       ↓
時空間コストマップに反映:
                       ↓
  t=0:  (x_0, y_0) 周辺を block
  t=1:  (x_1, y_1) 周辺を block
  ...
  t=11: (x_11, y_11) 周辺を block
                       ↓
                   STP4 探索
```

**重要**: 各歩行者の **不確実性** (Kalman の共分散) も block 半径に反映可能な設計。

## 5.5 性能とトレードオフ

| 項目 | 値 |
|------|-----|
| 全パイプライン (検出 + 追跡 + 予測) | 10 Hz でリアルタイム |
| 1 フレームあたり計算時間 (典型) | 10 – 30 ms (CPU only) |
| 同時追跡可能歩行者数 | 数十人 (実用上問題ない範囲) |

### 既知の限界

| 項目 | 内容 |
|------|------|
| 静止歩行者 | 動かない人を「障害物」と区別するのが難しい (どちらも STP4 に渡せば OK) |
| 物陰からの出現 | LiDAR が見えていないと検出不可 (物理限界) |
| 走る歩行者 | 軌道予測の線形仮定が崩れる場合あり |

## 5.6 設計判断の振り返り

| 判断 | 採用 | 理由 |
|------|------|------|
| 検出に Deep Learning を使うか | 使わなかった (RF) | データ量制約 + CPU 動作の要件 |
| 追跡に ByteTrack を採用 | ✓ | 2 段マッチングで安定 |
| 軌道予測に ONNX | ✓ | フレームワーク非依存・推論速度安定 |
| 検出器を Hesai 専用に再学習 | ✓ | センサー固有の点群密度に最適化 |

## 5.8 フレームレート整合 — 実測タイムスタンプ対応 (2026-06)

### 5.8.1 見つかった不整合

追跡 (Kalman) と軌道予測のサンプリングは、いずれも **固定フレームレートをハードコード** していた:

- Kalman の予測ステップ `dt = 1/track_fps`（10 Hz 前提 = 0.1 s）
- 予測の観測サンプリング stride = 4（10 Hz ÷ 4 = 学習レート 2.5 Hz を狙った値）

ところが実際の検出パイプラインは CPU/IO 負荷低減のため **約 5 Hz** で回っていた。結果として:

| 箇所 | 前提 | 実際 | 影響 |
|------|------|------|------|
| Kalman dt | 0.1 s | 0.2 s | 動作モデルが約 2 倍ズレ → 対応付けが不安定、ID 切替・分断が増える |
| 予測観測間隔 | 0.4 s (2.5 Hz) | 0.8 s (1.25 Hz) | 学習分布とズレ → 予測軌跡が時間・距離とも過大 |

### 5.8.2 対応

検出 1 フレームごとに **LiDAR タイムスタンプの実時刻差 `frame_dt`** を求め、追跡と予測の両方に流し込む:

```
frame_dt = stamp(t) − stamp(t-1)     // 0.01–1.0 s にクランプ
  ├─ Kalman:  tracks.set_dt(frame_dt)               // dt を実測値で駆動
  └─ 予測:    stride = round(0.4 s / frame_dt)       // 観測を ≈0.4 s 間隔へ動的整合
              pred_dt = frame_dt × stride
```

- **Kalman** は実レートに追従するため、5 Hz / 10 Hz / 変動のいずれでも動作モデルが正しい。
- **予測** は観測が常に学習レート (≈0.4 s = 2.5 Hz) に揃うため ONNX 入力分布が一致し、さらに予測 horizon (≈4.8 s) がプランナの計画 horizon と一致するので、既存のフレーム単位リサンプルがそのまま正しく機能する（プランナ側は無改修）。

### 5.8.3 状態と限界

- **実装・ビルド (コンパイル) 確認済み**。実機 / 録画リプレイでの効果検証は **今後の課題**。
- 観測サンプリングはフレーム数による「**だいたい 0.4 s**」整合 (index ベース)。タイムスタンプにジッタがある場合の厳密な等時刻サンプリングではない (履歴の各点に時刻を持たせて時刻補間する方式が厳密版だが、レートが安定なら近似で実用十分と判断)。
