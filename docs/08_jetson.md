# 08. Jetson 実機移行と並列高速化

[← 07. 学んだこと](07_lessons.md) | [目次に戻る →](../README.md)

---

> 開発 PC (x86_64) で組み上げたスタックを、ロボット搭載計算機 **Jetson AGX Orin (aarch64)** へ移行した記録。
> 「git clone しても動かない」原因の解消、2 台のマシンを **同一リポジトリ・同一手順** で運用するための互換基盤、
> そして Jetson 実機上での検出前段の **3 倍高速化 (100.5 → 33.3 ms)** まで。実施期間: 2026-07。

---

## 8.1 なぜ Jetson 移行か

これまでの開発・検証は x86_64 の開発 PC 上で行っていたが、実運用ではロボット搭載計算機 (Jetson AGX Orin, 12 コア CPU) 上で全パイプラインが動く必要がある。本パイプラインは GPU 不要の CPU-only 設計 ([05. 歩行者検出](05_pedestrian.md)) のため、移行のポイントは:

1. **ビルド・実行環境の移植性** — aarch64 でそのままビルド・起動できるか
2. **結果の互換性** — アーキテクチャが変わっても同じ検出結果が出るか
3. **実機での処理速度** — 開発 PC より遅い CPU で計画予算 (5 Hz = 200 ms) に収まるか

---

## 8.2 移行で露見した問題 — 「git clone しても動かない」

sim → real ([07. 学んだこと](07_lessons.md)) と同様、マシン → マシンにもギャップがあった。

| 問題 | 内容 | 対処 |
|------|------|------|
| 未コミット資産 | ビルドに必須のソース・設定が開発 PC のローカルにだけ存在 | 追跡下に追加。「clone だけで再現できるか」を移行で初めて検証できた |
| TF ツリー衝突 | launch 構成の重複により TF が二重配信 | launch を整理し配信元を一元化 |
| 実行時リンク | install 後に `LD_LIBRARY_PATH` を手動設定しないと起動しない | rpath を設定し、環境変数なしで起動可能に |
| パスのハードコード | `/home/<user>` 決め打ちが launch・スクリプトに散在 | env + 自動検出方式に置換 (§8.3 の運用ルール化へ) |
| ONNX Runtime の場所 | マシンごとに配置・アーキ (x86_64 / aarch64) が異なる | CMake の自動探索モジュールを新設 (§8.3) |

また移行を機に全コードレビューを実施し、検出・計画まわりの潜在バグ、フレーム座標契約、teleop フェイルセーフなどの安全性修正も行った。

> **一言**: 「1 台のマシンで動いている」状態は、環境がコードの欠陥を隠しているだけかもしれない。**2 台目への展開は移植性の回帰テスト** になる。

---

## 8.3 クロスマシン互換の枠組み

両マシンを同一手順で扱うため、管理の原則をルール化した:

| 何を | どこで管理 | 備考 |
|------|-----------|------|
| コード・設定・ドキュメント | **git** | マシン固有の値をコミットしない |
| ONNX モデル・データ (bag/bin/動画) | **クラウドストレージ (rclone)** | git には入れない |
| マシン差 (パス・アーキ) | **自動検出 + env 上書き** | ハードコード禁止 |

ONNX Runtime は「明示指定 → 環境変数 → システム標準パス → ホーム配下の最新版」の順で自動探索する規約に統一し、CMake・環境スクリプト・launch がすべて同じ順で解決。**どちらのマシンでもフラグなしで同じコマンドでビルド・起動できる**。

新マシンの立ち上げは 3 コマンドに集約した:

```
setup_machine.sh      # 一括セットアップ (冪等: apt / ORT / venv / モデル / ros2_ws / build)
check_env.sh          # 静的診断 (FAIL 時は修復コマンドを提示)
regression_check.sh   # ゴールデン回帰 (検出結果の互換性を実証)
```

### ゴールデン回帰 — 「互換」の定義を決める

x86_64 と aarch64 では libm (sin/cos 等) の実装が異なり、**浮動小数の結果はビット一致しない可能性がある**。そこで互換性の定義を「**許容誤差内で同一の検出結果**」と明文化した:

- 基準データ: 実機録画 92 フレームの検出結果 CSV をリポジトリにコミット
- 判定: 位置 0.05 m / スコア 0.05 の許容誤差 + フレーム毎の検出数一致 + ID 対応の一貫性
- 同一マシン内では完全に決定的 (diff 0.0000 を確認済み)
- 挙動を意図的に変えたときだけ基準を更新してコミット

> **一言**: 「同じ結果」は自明ではなく、**定義してテストで守るもの**。アーキテクチャ差を知らずにビット一致を要求すると、永遠に「壊れている」ように見える。

## 8.4 検出前段の並列高速化 — Jetson で 100.5 → 33.3 ms

### 8.4.1 背景 — 実機 CPU では 4 倍遅い

開発環境で 26.5 ms/frame だった検出→予測パイプライン ([06. 評価・結果](06_results.md) §6.8) が、Jetson 実機では **約 107 ms/frame** かかった。計画予算 200 ms には収まるものの余裕が薄いため、ボトルネックを再計測して高速化した (対象: 実機録画 92 フレーム、Jetson 12 コア)。

### 8.4.2 計測 — 主犯は「クラスタリング本体」ではなかった

段別計測の結果、狭義のクラスタリング (connected component) は **わずか 2.0 ms** で、重いのはその前後だった:

| 処理 | 平均 ms/frame | 割合 |
|------|---:|---:|
| グリッド構築 (2 種の map を別々に構築) | 24.9 | 27% |
| 適応クラスタリング (遠方再クラスタ + merge/link/split) | 31.0 | 34% |
| Hough 変換による壁フィルタ | 26.8 | 29% |
| クラスタリング本体 | 2.0 | 2% |
| 地面除去・ROI trim ほか | 7.1 | 8% |

> **一言**: 「クラスタリングが重い」という思い込みのまま並列化していたら、全体の 2% を最適化していた。**並列化の前にまず計測**。

### 8.4.3 実装 — アルゴリズム改善 × 決定的並列化

OpenMP による並列化と、直列でも効くアルゴリズム改善を併用した。**すべて「出力が変わらない」設計** (順序保持・race 回避) を条件にしている。

| 対象 | 主な手段 | 効果 |
|------|---------|-----:|
| 壁フィルタ (Hough) | 三角関数の恒等式で角度ループを積和 2 回に軽量化、O(N²) の幅カウントをソート + 二点ポインタ O(N log N) 化、クラスタ単位で並列化 | 26.8 → 1.9 ms (**15.6x**) |
| グリッド構築 | 2 回走査していた 2 種の map を **1 回の点群走査で同時構築** (counting-sort 方式)、行単位で並列組み立て | 24.9 → 8.7 ms (2.9x) |
| 適応クラスタリング | merge/split をクラスタ並列、link は「並列で辺収集 → 直列で union-find」に分離して race 回避 | 31.6 → 17.1 ms (1.8x) |
| その他 | map の値渡し → const 参照 (deep copy 除去)、z シフトと ROI trim の 1 パス融合 | 数 ms |

### 8.4.4 結果

| 指標 | 改善前 | 改善後 | 倍率 |
|------|---:|---:|---:|
| 検出前段 (detect) | 100.5 ms | **33.3 ms** | **3.0x** |
| 検出→予測 全体 | 106.9 ms | **39.6 ms** | 2.7x |
| 全体 p95 | 113.7 ms | 45.1 ms | 2.5x |
| 全体 max | 116.4 ms | 51.1 ms | 2.3x |

- **出力は改善前とビット単位で完全一致** (検出 CSV・予測 CSV を diff で確認、クラスタ数系列も全フレーム一致)
- 1 スレッド実行でも出力一致・detect 57.8 ms — **アルゴリズム改善だけで 1.7x**、並列化でさらに 1.7x
- OpenMP 未検出環境では自動的に直列動作 (ビルドは通る)

### 8.4.5 「決定的な並列化」にこだわった理由

並列化で結果が変わると、§8.3 のゴールデン回帰が使えなくなり、「速くなったが正しいか分からない」状態になる。辺収集と union の分離、keep フラグ方式による順序保持など、**等価性を証明できる形** に実装を制約した。結果として回帰テストがそのまま高速化の検証にも使えた。

> **一言**: 高速化の価値は「速さ × 信頼」。**ビット一致という強い証拠** があれば、最適化を恐れず入れられる。

---

## 8.5 到達点と残課題

**到達点**:

- Jetson 実機で スタンドアロン・ROS 2 の両構成がビルド・動作
- 開発 PC / Jetson の双方でフラグなし同一手順のビルド・起動・回帰テスト
- Jetson 実機で検出→予測 39.6 ms/frame — **計画予算 200 ms に対し約 5 倍の余裕**

**残課題**:

- さらなる高速化 (20 ms 級) には遠方グリッド再構築の差分化やデータ構造の flat 化が必要 (ビット等価とのトレードオフ)
- 一部 Python スクリプトに開発 PC のパスが残存 (置換予定)
- ONNX Runtime 更新は両マシン同時に行う運用ルールで管理
